# 《房间公理》产品设计文档

**版本：** 0.1 交接基线  
**日期：** 2026-06-22  
**状态：** 核心方向锁定；待实现与试玩验证

---

## 1. 产品摘要

《房间公理》是一款格状、回合制、无反应速度要求的推理游戏。它借用规则怪谈的叙事语气，却遵守比普通怪谈更严格的逻辑契约：

- 玩家进入房间前，所有机械规则已完整公开；
- 房间中每个格子的内容在开局前已经固定；
- 调查行为只揭示客观事实，不会产生新规则；
- 新揭示的信息只会缩小候选世界集合，不会推翻已经正确完成的推理；
- 发布关卡必须经过“零猜测”和最终危险布局唯一性的机器验证。

经典扫雷用数字描述相邻地雷数量；本项目用物件与物件之间的固定关系描述隐藏的“访客”。酒瓶、垃圾桶、镜子、床、门牌、监控等物证都是不同语义的线索符号。

### 1.1 核心卖点

1. **推导式恐怖**：玩家不是被跳脸告知怪物存在，而是从陈设关系亲手算出某处必有“访客”。
2. **固定世界的安全感**：规则怪谈的氛围与严格逻辑题的公平性共存。
3. **可解释的零猜测**：每一步都能给出公开规则、已知物证和集合关系构成的证明。
4. **数据驱动的内容量产**：规则模板、关卡 JSON、精确求解器和人类推理器共同支撑生产。

### 1.2 目标平台与玩家

MVP 目标为桌面与平板浏览器；手机可玩但不是首轮体验基准。目标玩家包括：

- 喜欢扫雷、数独、非图、逻辑矩阵等纯推理游戏的玩家；
- 喜欢规则怪谈、密室、都市怪谈氛围，但反感作者临时改规则的玩家；
- 愿意阅读短规则并进行集合交叉推理的玩家。

### 1.3 非目标

MVP 不做：实时追逐、动作操作、倒计时、复杂剧情分支、联网、UGC 分享平台、付费系统、3D 场景、随机 Roguelite 规则增殖。

## 2. 已达成一致的设计结论

以下内容视为当前基线，除非通过新的架构/产品决策记录正式修改。

### 2.1 只推进固定规则方向

本项目不采用“每次交互后出现一条未来规则”的动态规则玩法。动态规则可作为未来独立项目，不与当前核心循环混合。

### 2.2 规则与揭示严格分层

- **规则（Axiom）**：开局公开、全局有效、语义不可变。
- **事实（Observation）**：某格实际是什么物件；通过调查逐步揭示。
- **笔记（Annotation）**：玩家对未知格的暂时判断，不改变世界。
- **结论（Deduction）**：由规则与事实推出的必然安全或必为访客。

### 2.3 禁止隐藏规则与追溯矛盾

关卡不得依赖：

- 未公开的邻域定义；
- 文案双关或作者常识；
- “原来这条规则只在某条件下成立”的事后补充；
- 后续揭示使之前基于公开信息的正确结论变错；
- 以“本题唯一解”本身作为玩家必须使用的隐藏线索。

### 2.4 中间多解允许，猜测点禁止

候选世界集合可以暂时包含多个完整布局。合格状态必须满足至少一项：

- 存在一个在所有候选世界中都不是访客的未调查格；
- 存在一个在所有候选世界中都是访客的未确定格；
- 危险布局已经唯一，可提交结论。

### 2.5 最终唯一的是危险布局

安全陈设的完整分布可以不唯一。玩家的目标是唯一确定“访客”所在格，而不是复原每一处空地或物件。

### 2.6 两套求解器共同验收

- **精确求解器**负责正确性：一致性、强制格、最终唯一、反例搜索。
- **人类推理器**负责可读性：使用有限的、明确的推理模板产生证明。

若精确求解器能证明，但人类推理器无法用允许的推理模板推进，则关卡标为“机器可解、人类解释缺失”，不可进入正式内容。

## 3. 核心体验与游戏循环

### 3.1 玩家幻想

玩家是夜间调查员、清扫员或临时住客，收到一份冷静、精确、略显不祥的房间守则。房间被划分为可调查区域。某些区域中存在伪装成陈设或人的“访客”。玩家必须在不直接触碰访客的前提下，通过物件关系确定其位置。

### 3.2 单局循环

1. 阅读本局全部规则及精确定义。
2. 查看初始可见物证和固定地形。
3. 交叉比较规则，推出一个必然安全格或必为访客格。
4. 调查安全格，揭示物件；或对危险格做访客标记。
5. 新事实加入证据集，候选世界集合缩小。
6. 重复，直到危险布局唯一并提交结论。
7. 结算展示用到的推理链、无效调查数、提示使用和可选最短解。

### 3.3 玩家动作

| 动作 | 作用 | 风险 |
|---|---|---|
| 调查未知格 | 揭示该格客观物件 | 若为访客，本局失败 |
| 标记访客 | 记录玩家认为该格是访客 | 不立即判定，可撤销 |
| 标记安全 | 铅笔式笔记，表示计划调查 | 不改变规则或事实 |
| 查看规则范围 | 高亮正交邻域、邻接域等 | 无 |
| 请求提示 | 给出一个可解释的下一步，不直接泄露全解 | 影响评分 |
| 提交结论 | 检查全部访客标记 | 错误时 MVP 允许继续修正 |
| 重置/撤销笔记 | 回到初始或修改笔记 | 调查事实不可撤销，重置可重开 |

### 3.4 成功与失败

- **成功**：标记数量符合公开总数，且所有标记与目标危险布局一致。
- **失败**：调查到访客。
- 错误标记不立即失败，以避免把笔记操作变成试错探测器。
- MVP 提交错误只提示“当前结论不成立”，不指出哪一格错误。

## 4. 语言与空间语义

上一版文本测试暴露了“周围八格”对边缘格的误导。当前项目必须统一使用以下术语。

### 4.1 坐标

- 列使用大写字母 A、B、C……
- 行使用从 1 开始的数字。
- `B3` 表示 B 列第 3 行。

### 4.2 正交邻域

与目标格共享一条边、且位于棋盘内的格子集合。

- 角落：2 格；
- 边缘非角落：3 格；
- 内部：4 格。

### 4.3 邻接域

与目标格共享边或角、且位于棋盘内的格子集合。

- 角落：3 格；
- 边缘非角落：5 格；
- 内部：8 格。

UI 不得只写“附近”或“周围八格”。规则卡必须包含术语、数量词和可视范围按钮；指向某格时直接高亮实际范围。

### 4.4 数量词

- `恰好 N`：不少于也不多于 N。
- `至少 N`：N 或更多。
- `至多 N`：N 或更少。
- `没有`：恰好 0。

规则怪谈式文案可放在标题或风味文本中，但机械语义必须单独显示。

示例：

> 清扫员只会给每只空酒瓶留一个位置。
>
> **精确规则：每个酒瓶的正交邻域中，恰好有 1 个垃圾桶。**

## 5. MVP 物件与基础规则库

### 5.1 格子类型

| 类型 | 是否安全调查 | 主要功能 |
|---|---:|---|
| 空地 | 是 | 不提供局部约束，但排除其他类型 |
| 酒瓶 | 是 | 通过正交邻域约束垃圾桶或访客 |
| 垃圾桶 | 是 | 提供安全区或数量约束 |
| 镜子 | 是 | 对邻接域内访客数量提供约束 |
| 访客 | 否 | 本局危险目标 |

### 5.2 MVP 允许的规则模板

1. **全局数量**：某类型全局恰好/至少/至多 N 个。
2. **逐主体计数**：每个 A 的某邻域内，B 恰好/至少/至多 N 个。
3. **逐主体排除**：每个 A 的某邻域内没有 B。
4. **唯一目标交集**：当 B 全局唯一，多个 A 均要求邻接 B，可通过候选交集定位 B。
5. **总量闭合**：已确认数达到总数后，其余候选安全；剩余候选数等于剩余目标数时全部为目标。
6. **局部集合差**：两个计数邻域重叠时，用集合差推出额外安全/危险格。

MVP 不加入真假话、自指、条件反转、时间轴、视线遮挡、连通性和移动实体。

### 5.3 风味文本原则

- 风味文本不得包含额外机械信息。
- 同一机械规则在所有关卡中使用统一图标和标准句式。
- 可以变化的是叙事包装，例如“客房”“诊所”“仓库”；机械关键词必须保持一致。

## 6. 零猜测与唯一性设计契约

### 6.1 候选世界

令 `K_t` 表示第 t 步时所有符合以下条件的完整世界：

- 满足全部开局规则；
- 满足所有已揭示事实；
- 不把已调查安全格设为访客；
- 与已确认的目标总数一致。

每次合法调查只添加事实，因此：

`K_(t+1) ⊆ K_t`

### 6.2 强制安全与强制危险

- 格 c **强制安全**：在 `K_t` 的每个世界中，c 都不是访客。
- 格 c **强制危险**：在 `K_t` 的每个世界中，c 都是访客。

### 6.3 关卡验收

从初始状态开始重复：

1. 若候选世界为空，关卡矛盾，失败。
2. 若所有候选世界的访客集合相同，最终危险布局唯一，通过唯一性阶段。
3. 否则必须存在人类推理器可证明的安全调查或危险标记。
4. 将所有当前可证明的安全格按目标棋盘揭示，将可证明危险格标记，继续。
5. 若精确求解器存在强制格，但人类推理器无证明，标记为解释缺口。
6. 若两者都无法推进且危险布局未唯一，判定为猜测点，关卡失败。

### 6.4 路径独立性

事实单调增加，候选集合只缩小。某格一旦在当前候选集合中被证明安全，后续事实不会使它变为危险。因此玩家以任意顺序调查当前所有已证明安全格，都不会破坏可解性。

### 6.5 不把“唯一解”当线索

玩家界面可以承诺本关通过零猜测验证，但具体推理不得使用“因为设计者保证唯一，所以 X 必须成立”。提示必须引用规则、事实和集合关系。

## 7. 基准样例：客房 04

### 7.1 开局规则

1. 房间中恰有 1 个垃圾桶。
2. 房间中恰有 2 名访客。
3. 每个酒瓶的正交邻域中，恰好有 1 个垃圾桶。
4. 每个垃圾桶的邻接域中，没有访客。
5. 每面镜子的邻接域中，恰好有 1 名访客。
6. 每个酒瓶的正交邻域中，没有访客。

### 7.2 目标棋盘

|   | A | B | C | D |
|---|---|---|---|---|
| 1 | 空地 | 酒瓶 | 镜子 | 访客 |
| 2 | 酒瓶 | 垃圾桶 | 酒瓶 | 空地 |
| 3 | 镜子 | 空地 | 镜子 | 空地 |
| 4 | 空地 | 访客 | 空地 | 空地 |

初始只揭示 `B1`、`A2`、`C2` 三只酒瓶。

### 7.3 推荐推理链

1. 三只酒瓶的正交邻域交集只有 `B2`；垃圾桶全局唯一，所以 `B2` 必为垃圾桶且可安全调查。
2. 垃圾桶的邻接域无访客，因此 `A1、C1、A3、B3、C3` 安全；酒瓶规则还推出 `D2` 安全。
3. 调查 `C1` 得镜子。其邻接域内唯一仍可为访客的位置是 `D1`，所以 `D1` 必为访客。
4. 调查 `A3` 得镜子。其邻接域要求恰好一名访客，因此 `A4、B4` 中恰有一个。
5. 调查 `C3` 得镜子。结合总访客数与前两面镜子，推出第二名访客只能在 `B4`。
6. 标记 `D1、B4`，提交成功。

### 7.4 候选危险布局收缩

- 初始：15 种；
- 揭示 `C1=镜子`：5 种；
- 揭示 `A3=镜子`：2 种；
- 揭示 `C3=镜子`：1 种。

这个样例用于 UI 原型和首批求解器回归测试。

## 8. UI 对推理的支持

产品不能把工作记忆负担误当难度。界面必须提供：

- 始终可见的规则列表；
- 规则机械语义与风味文本分层；
- 点击规则后高亮其邻域类型；
- 坐标、物件图标和文字标签；
- 访客标记与安全笔记；
- 已揭示证据日志；
- 可选提示证明；
- 开发者模式下显示候选数、强制格和验证状态。

默认玩家模式不得自动高亮强制安全/危险格；这属于提示或开发者信息。

## 9. 难度模型

单纯扩大棋盘不是主要难度来源。建议按以下指标评分：

1. **推理波次**：从开局到唯一危险布局需要多少轮新揭示。
2. **同时变量数**：某一步需要同时考虑的候选格数量。
3. **规则交叉数**：一个结论引用几条规则和几项事实。
4. **集合运算深度**：是否需要邻域交集、差集或总量闭合。
5. **可选安全格数量**：过多会变成扫清空地；只有一个会显得线性。
6. **无信息调查比例**：安全但不产生新约束的格子占比。
7. **证明长度**：人类推理器生成的最短证明节点数。
8. **视觉扫描负担**：同类图标数量、规则范围重叠和棋盘尺寸。

### 9.1 关卡分层

- 教学：4×4，2 名访客，单一邻域，直接计数。
- 初级：4×4/5×5，交集和总量闭合。
- 中级：5×5，多个物件类型，局部集合差。
- 高级：未来加入行列、距离或视线，但仍要求可解释和零猜测。

## 10. 内容生产流程

### 10.1 MVP 阶段

先手工设计目标棋盘和规则组合，再由验证器验收。首批 10 关应覆盖每种基础推理模板，而不是追求数量。

### 10.2 量产阶段

1. 选择规则模板集合和目标难度区间。
2. 随机或反向构造满足规则的完整棋盘。
3. 选择初始揭示物证。
4. 精确求解器检查一致性和危险布局唯一性。
5. 零猜测验证器模拟推理闭包。
6. 人类推理器生成证明，检查模板覆盖。
7. 删除冗余初始揭示；每次删除后重新验证。
8. 计算难度、信息增益和重复度。
9. 人工审阅文案、视觉布局和解题节奏。

### 10.3 关卡发布门槛

- Schema 有效；
- 目标棋盘满足规则；
- 初始状态至少有一个可解释动作；
- 无猜测点；
- 最终访客集合唯一；
- 无规则歧义；
- 最短证明不超出目标难度；
- 自动测试快照通过；
- 人工试玩无误导。

## 11. 叙事与美术方向

### 11.1 叙事原则

恐怖来自秩序过于精确，而不是规则随意耍赖。管理方、清扫员或“住客手册”以行政文本口吻描述异常：

- “请保证每只空酒瓶都能被唯一的清扫容器照看。”
- “镜面中只能出现一位未登记住客。”
- “垃圾桶附近不应出现任何会呼吸的东西。”

精确规则紧随其后，避免玩家猜文案隐喻。

### 11.2 视觉原则

- 低饱和、纸张与监控终端混合质感；
- 危险不依赖血腥表现；
- 已知物证清晰、图标差异大；
- 未知格保持不安但可读；
- 颜色只作辅助，必须同时使用形状和文字。

### 11.3 音频原则

MVP 可无音频。后续加入环境声和揭示反馈，但不得用声音承载唯一机械信息。

## 12. 教学设计

第一关只教三个概念：

1. 规则开局固定；
2. 调查揭示物件；
3. “恰好”与邻域图示。

教学提示应要求玩家说出或选择理由，例如：

> 为什么 B2 安全？
>
> A. 因为它离入口最近  
> B. 因为三只酒瓶的垃圾桶候选交集只有 B2  
> C. 因为第一步总是安全

正确答案 B。这样能验证玩家理解的是逻辑，而不是记住安全序列。

## 13. 试玩与数据指标

MVP 试玩重点不是留存，而是验证认知契约。

### 13.1 核心观察

- 玩家是否把规则与揭示事实区分开；
- 是否理解边缘邻接域只有棋盘内格子；
- 每次调查前能否说明“为何必然安全”；
- 是否误读单向蕴含为双向关系；
- 规则卡是否足够快地用于比较；
- 玩家是否因可选安全格过多而机械扫格；
- 提示证明是否令人信服。

### 13.2 建议事件

- case_started
- rule_scope_opened
- cell_inspected
- guest_mark_changed
- safe_note_changed
- hint_requested
- conclusion_submitted
- case_failed
- case_completed

不记录个人身份信息。开发阶段可记录每步候选数和证明类型。

### 13.3 成功阈值

- 80% 以上测试者能在第一关解释第一步理由；
- 边缘邻域误解率低于 10%；
- 非提示完成者中，因纯猜测造成的失败为 0；
- 至少 60% 测试者认为“结论是自己推出来的”，而非“试出来的”。

## 14. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 自然语言歧义 | 玩家把规则误读成不同逻辑 | 机器句式、图示范围、术语表 |
| 单向蕴含误读 | 推出不存在的镜子数量等 | 教学反例、箭头式规则展示 |
| 数学可解但人类难懂 | 公平性名义成立，体验仍差 | 双求解器、证明长度门槛 |
| 关卡过度线性 | 像按提示走流程 | 每波保留 2-4 个安全选择，控制信息增益 |
| 安全空格过多 | 变成机械清扫 | 限制无信息格，自动展开可选但谨慎使用 |
| 规则模板重复 | 换皮感强 | 后续逐步加入行列、距离、视线等可解释模板 |
| 求解性能 | 编辑器和提示卡顿 | Worker、位掩码、缓存、查询预算 |
| 怪谈氛围压过可读性 | 看不清物件与规则 | 视觉层级优先，氛围不牺牲对比度 |

## 15. MVP 范围

### 15.1 必须有

- 4×4 与 5×5 棋盘；
- 五种格子类型；
- 六类基础规则/推理；
- 10 个手工关卡；
- 完整规则 briefing；
- 调查、两类标记、提示、重置、提交；
- 精确求解器、人类推理器、关卡验证 CLI；
- 开发者验证层；
- 响应式桌面/平板 UI；
- 自动化测试和静态部署。

### 15.2 应该有

- 关卡选择与成绩；
- 可访问键盘操作；
- 提示证明图；
- 候选数和强制格的内部调试面板；
- 关卡 JSON 热加载（开发模式）。

### 15.3 暂不做

- 程序化关卡对玩家开放；
- 关卡编辑器完整 UI；
- 云存档；
- 多语言；
- 复杂剧情演出；
- 商业化。
