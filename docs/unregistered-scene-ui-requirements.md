# 《未登记现场》UI 需求文档

版本：v0.1  
用途：给主题包装、原画、UI 设计和后续前端实现阅读  
基线：不改变 Room Axioms 当前核心玩法、规则判定、求解器、证明系统和关卡验收逻辑

## 1. 文档目标

本文档描述《未登记现场》主题包装下需要有哪些 UI 模块、每个模块如何交互、哪些视觉资产需要准备，以及新增“日式视觉小说对话系统”后如何与现有格子推理玩法衔接。

核心原则：

- 只做主题包装与表现层增强，不改变底层玩法。
- 所有可用于解题的信息仍然只来自公开规则、已调查事实和系统提示。
- 剧情文本、角色表情、音效、背景图、异常美术、现场记录都不能暗示异常格位置。
- 玩家身份从“抽象解谜者”包装为“非常规赔案现场公估师”。
- 规则从“房间公理”包装为“现场定则”。
- 危险目标从“访客/目标”包装为“异常区域”。

## 3. 总体页面结构

桌面端建议保持三栏工作台：

1. 顶部栏：品牌、案卷名、案卷选择、进度、搭档复核、重置。
2. 左侧栏：现场定则。
3. 中央区：现场平面图。
4. 右侧栏：现场登记记录、玩家标注、提交按钮。
5. 弹层：提示、成功、失败、视觉小说对话。

移动端建议保持底部/顶部标签切换：

- 定则
- 现场图
- 登记

视觉小说对话在移动端应全屏或近全屏显示，避免半身像和文本框挤压棋盘。

## 4. UI 模块需求

### 4.1 顶部栏：案卷与行动区

目的：

- 告诉玩家当前正在处理哪一宗赔案。
- 提供案卷切换、搭档复核、重置入口。
- 展示关键进度，但不泄露答案。

主要内容：

- 项目标题：《未登记现场》。
- 英文或副标题：Unregistered Scene / Abnormal Claim Survey。
- 当前案卷名，例如：案卷 04 · 客房清扫记录。
- 案卷选择下拉框。
- 已标注异常数量 / 目标异常总数。
- “搭档复核”按钮。
- 重置按钮。

交互：

- 切换案卷后重置当前局状态，载入新案卷。
- 点击“搭档复核”打开提示弹窗或触发搭档对话。
- 点击重置清空调查、标记、提示和结论状态。

原画/视觉要求：

- 顶部栏像保险公司内部系统或现场指挥软件。
- 品牌标识可以是公司部门徽记、文件章、扫描标记，不要像奇幻法阵。
- 案卷选择像档案编号或案件管理系统。

### 4.2 现场定则面板

目的：

- 展示开局前公开的全部规则。
- 玩家点击某条定则后，现场图高亮该定则会检查的格子。

主要内容：

- 面板标题：现场定则。
- 规则卡片列表。
- 每条规则包含编号和白话规则句。

交互：

- 点击规则卡片：选中/取消选中。
- 选中后：现场图高亮规则生效范围。
- 再次点击同一规则：取消高亮。

文案要求：

- 避免抽象术语堆叠。
- “正交邻域”显示为“上下左右邻格”。
- “邻接域”显示为“周围一圈”。
- 规则句应像人话，例如：
  - “异常区域不在应急灯的周围一圈。”
  - “每具尸体的上下左右邻格，恰好有 1 个损坏终端。”
  - “本现场共有 3 处异常区域。”

原画/视觉要求：

- 像搭档口述后被系统记录下来的“定则条目”。
- 可使用记录卡、扫描条、编号章、档案纸质纹理。
- 选中态要清楚，但不能让玩家误以为被高亮格就是答案。

### 4.3 现场平面图模块

目的：

- 承载主要游戏操作。
- 玩家在这里调查安全区域、标注异常区域、标注可勘察区域。

格子状态：

- 未知：信息遮蔽区域，只显示坐标、边界和统一遮蔽。
- 已调查安全：显示内容图标和登记标签。
- 标注异常：显示玩家的异常标注，不揭示真实内容。
- 标注可勘察：显示玩家的安全笔记，不揭示真实内容。
- 规则高亮：显示当前定则影响范围。
- 提示高亮：显示搭档复核指出的可行动格。
- 开发者高亮：仅维护者模式，不进入玩家体验。

交互：

- 当前工具为“勘察”：点击未知格，若安全则揭示；若异常则失败。
- 当前工具为“标注异常”：点击未知格，切换异常标注。
- 当前工具为“标注可勘察”：点击未知格，切换可勘察标注。
- 右键未知格：快速循环标注状态。
- 鼠标悬停：底部说明该格当前状态。
- 键盘方向键：在格子间移动焦点。
- Enter/Space：执行当前工具。

重要限制：

- 未知格的视觉差异不得暗示内容类型或异常位置。
- 已调查安全格可以显示具体物件图标。
- 异常格在正常游玩阶段不得直接显示真实形态。
- 成功结算或失败演出可以有限展示异常影像。

原画/视觉要求：

- 现场图应像 CAD 图、消防图、法证平面图或保险勘察软件。
- 未知遮蔽可以是统一雾化、噪声遮罩、黑色扫描块、红线封锁区。
- 格子边界必须清晰、可点击，不要为了氛围牺牲可读性。
- 图标需要统一、扁平、可远距离识别。

### 4.4 工具栏

目的：

- 让玩家选择当前点击格子的操作模式。

工具：

- 受控勘察：调查一个未知区域。
- 标注异常：记录该区域可能是异常坐标。
- 标注可勘察：记录该区域可安全调查。

交互：

- 点击工具按钮切换模式。
- 当前模式在现场图标题区或工具栏中明确显示。
- 标注工具只改变玩家笔记，不改变真实状态。

视觉要求：

- 工具按钮使用熟悉图标：
  - 勘察：放大镜、扫描框。
  - 异常标注：警示旗、禁入标。
  - 可勘察标注：圆圈、确认章。
- 按钮要像调查软件工具，而不是战斗技能。

### 4.5 现场登记记录面板

目的：

- 显示已经调查出的事实。
- 显示玩家当前标注。
- 提供提交入口。

主要内容：

- 已登记区域列表：坐标 + 内容类型。
- 玩家标注：异常坐标列表、可勘察列表。
- 提交现场登记图按钮。

交互：

- 调查安全格后，记录追加到列表顶部或底部。
- 标记变化实时更新玩家标注列表。
- 点击提交后进入成功/失败校验。

提交失败要求：

- 可以提示“现场登记图未通过复核”。
- 不得指出具体哪一格错。
- 不得泄露异常实际坐标。

视觉要求：

- 像保险公估记录、清点表、现场登记表。
- 可以使用表格、印章、案件编号、时间戳。

### 4.6 搭档复核 / 提示弹窗

目的：

- 给出当前状态下一个可证明的安全或异常结论。
- 用尽量少的文本解释依据。

触发：

- 顶部“搭档复核”按钮。
- 未来可由视觉小说对话系统承载一次性教学提示。

内容：

- 结论：例如“B3 可以勘察”或“C2 应标注为异常”。
- 用到的信息：列出相关现场定则和已登记事实。
- 简短原因：解释为什么这个结论必然成立。

交互：

- 打开后阻塞棋盘操作。
- 关闭后回到现场图。
- 可高亮结论涉及的格子。

角色包装：

- 提示可以表现为搭档说话，而不是系统冷冰冰弹窗。
- 但搭档不能直接报最终答案，也不能说出不可由当前事实证明的信息。

### 4.7 成功结算弹窗

目的：

- 告诉玩家现场登记通过。
- 生成冷静、程序化的保险结案摘要。

内容：

- 成功标题：现场登记图已通过复核。
- 摘要：已登记区域数、异常标注数、提示次数、勘察次数。
- 自动报告：人员伤亡、财产损失、异常直接损失是否成立、赔案状态。
- 操作：关闭、重新调查。

视觉要求：

- 像结案报告或处置移交通知。
- 可以在成功后有限展示异常影像或后续处置重建图。
- 不要在结算前展示异常真身。

### 4.8 失败弹窗

目的：

- 告诉玩家任务失败，但不泄露完整答案。

触发：

- 玩家主动调查异常区域。
- 提交现场登记图错误。

内容：

- 灾害或复核失败说明。
- 不指出异常真实位置。
- 操作：重新调查。

视觉要求：

- “调查到异常”可以有快速灾害演出。
- “提交错误”应更像复核退回，而不是灾害。
- 失败演出可以强烈，但不应成为可用于反推答案的信息。

### 4.9 维护者验证层

目的：

- 给开发者检查候选布局、强制安全、强制异常、证明链等。
- 玩家模式完全不可见。

交互：

- 只在开发者模式显示。
- 可显示目标布局、强制结论、候选数量、proof lines。

视觉要求：

- 可保持简洁工程面板。
- 不需要原画投入。
- 不得混入玩家界面。

## 5. 新增模块：日式视觉小说对话系统

### 5.1 模块目标

新增一个视觉小说式对话层，用于承载角色关系、案卷导入、教学说明、搭档提示包装和结案余韵。

它是表现层和叙事层，不改变核心解谜规则。

### 5.2 基本 UI 组成

视觉小说对话层包含：

- 背景图：办公室、现场入口、现场平面图背景、结算室等。
- 角色半身像：主角、搭档、可选配角。
- 对话框：底部横向文本框。
- 角色名牌：显示当前说话人。
- 文本内容：支持逐字显示，也可一键显示整句。
- 继续提示：点击、Enter、Space 进入下一句。
- 跳过按钮：跳过当前对话段。
- 自动播放按钮：可选。
- 对话记录按钮：查看本段已读文本。
- 关闭/进入现场按钮：进入玩法界面。

### 5.3 角色半身像演出

最低需求：

- 每个主要角色至少 1 个默认立绘。
- 搭档至少 4 个表情：
  - 平静
  - 思考
  - 提醒
  - 紧张
- 主角至少 3 个表情：
  - 平静
  - 判断
  - 受挫/警觉

演出能力：

- 左右站位。
- 当前说话人高亮，非说话人轻微压暗。
- 表情切换。
- 轻微入场/退场。
- 轻微震动用于异常灾害或突发提示。
- 不要求复杂 Live2D；静态半身像 + 表情差分即可。

视觉要求：

- 日式视觉小说可读性优先，半身像不能遮挡关键 UI。
- 角色设计应偏“职场异常调查”而不是战斗冒险。
- 搭档可以更有辨识度，但不要让其视觉元素暗示答案或规则真假。

### 5.4 对话触发点

建议触发以下对话段：

1. 案卷开始前
   - 介绍事故、职业目标、异常禁忌。
   - 结束后进入现场图。

2. 首次进入现场图
   - 搭档解释“现场定则已整理完毕”。
   - 简短提示玩家先看规则。

3. 首次选中规则
   - 教学：选中定则可以查看生效区域。

4. 首次调查安全格
   - 教学：安全区域会被登记，事实不会改变。

5. 首次标注异常
   - 教学：标注只是笔记，异常不能主动调查。

6. 请求提示时
   - 可用对话框包装搭档复核。
   - 对话内容必须来自 proof-backed hint。

7. 调查到异常时
   - 快速失败演出。
   - 不泄露其他异常位置。

8. 提交成功后
   - 搭档短评 + 保险结案摘要。

9. 提交失败后
   - 系统复核退回。
   - 不指出错误格。

### 5.5 对话内容边界

允许：

- 案件氛围。
- 职业职责。
- 操作教学。
- 搭档根据系统 hint 说出当前可证明结论。
- 成功后的结案摘要。

禁止：

- 对话暗示某个未调查格的真实内容。
- 表情、语气、停顿、镜头、音效暗示异常坐标。
- 对话选择影响规则、布局或候选现场。
- 搭档给出错误规则。
- 玩家通过反复触发失败对话枚举答案。

### 5.6 对话系统交互

输入：

- 鼠标点击：推进一句。
- Enter / Space：推进一句。
- Esc：打开跳过/关闭确认，不能误关重要教学。
- Ctrl 或长按：快进已读文本，可选。

状态：

- 对话中：棋盘不可操作。
- 对话结束：回到原面板和原焦点。
- 提示对话结束：高亮提示格。
- 失败对话结束：进入失败弹窗或重试入口。

可访问性：

- 文本必须完整显示，不只依赖语音。
- 逐字播放可关闭。
- 角色名、表情变化不应是必要信息。
- 移动端可隐藏非说话人半身像，确保文本可读。

### 5.7 对话数据建议

可以新增独立叙事数据，不进入 solver/proof：

```ts
type DialogueLine = {
  id: string
  speaker: 'protagonist' | 'partner' | 'system' | 'client'
  text: string
  portrait?: string
  expression?: string
  position?: 'left' | 'right' | 'center'
  background?: string
  sfx?: string
  waitMs?: number
}

type DialogueScene = {
  id: string
  trigger:
    | 'caseIntro'
    | 'firstRuleSelect'
    | 'firstSafeInspect'
    | 'firstAnomalyMark'
    | 'hint'
    | 'failure'
    | 'success'
  lines: readonly DialogueLine[]
  skippable: boolean
}
```

重要：对话数据只能引用展示资产和文案，不能改变 puzzle target、rules、observations 或 solver state。

## 6. 原画资产需求清单

### 6.1 角色

主角：非常规赔案现场公估师。

- 半身像：默认、判断、警觉。
- 气质：谨慎、程序化、职业调查员。
- 道具：平板、档案夹、现场勘察设备、耳麦。

搭档：定则员 / 定则感知者。

- 半身像：默认、思考、提醒、紧张。
- 气质：冷静但有人味，像能感知规则的同事，不是神秘占卜师。
- 道具：耳机、录音笔、规则记录终端、公司制服元素。

可选角色：

- 保险公司系统语音/操作员。
- 客户方联系人。
- 专业处置单位人员。

### 6.2 场景背景

基础背景：

- 非常规赔案部门办公室。
- 现场外封锁线。
- 现场平面图工作台。
- 结案报告界面。

首案《别按那个键》背景：

- 小型交互设备公司办公室。
- 测试间外部。
- 监控室或设备仓。
- 成功结算后的黑盒处置影像。

### 6.3 UI 视觉资产

- 项目 logo / 部门标识。
- 案卷卡片样式。
- 现场图边框和坐标样式。
- 未知遮蔽纹理。
- 勘察区选中、高亮、异常标注、可勘察标注样式。
- 现场定则卡片。
- 现场登记表样式。
- 结案摘要样式。
- 视觉小说对话框、名牌、继续提示、跳过按钮。

### 6.4 图标

规则对象图标：

- 应急灯。
- 测试终端。
- 尸体。
- 档案柜。
- 病床。
- 设备。
- 空区。
- 产品原型。
- 会议桌。
- 快递包装。

操作图标：

- 受控勘察。
- 标注异常。
- 标注可勘察。
- 搭档复核。
- 提交现场登记图。
- 重置案卷。

### 6.5 异常素材

首案黑盒：

- 未直接揭示版：只用于结算或失败演出，不用于正常棋盘推理。
- 成功结算重建图：可以展示哑黑盒和红色按键。
- 失败演出素材：短暂、冲击强，但不能显示其他答案。

## 7. 信息层级与屏幕优先级

玩家每一刻最需要看到：

1. 当前现场图。
2. 当前工具模式。
3. 现场定则。
4. 已登记事实。
5. 玩家标注。
6. 搭档复核入口。
7. 提交入口。

视觉小说对话可以覆盖屏幕，但不应在玩家需要推理时长期占据空间。进入正式推理后，对话应收起为提示/复核系统。

## 8. 移动端要求

移动端保留三个主面板：

- 定则
- 现场图
- 登记

移动端对话系统：

- 对话框固定底部。
- 当前说话人半身像优先显示。
- 非说话人可隐藏或缩小。
- 按钮必须足够大，避免误触。
- 文本不能被刘海屏、安全区、浏览器工具栏遮挡。

## 9. 不改核心玩法的实现边界

主题包装阶段不改：

- Puzzle Schema v1 的解题语义。
- solver / proof / oracle 的算法。
- 关卡 target。
- 规则判定。
- 无猜测验证。
- 玩家提交校验。
- 失败不泄露答案的原则。

可以新增或修改：

- UI 文案。
- 展示组件。
- 视觉主题。
- 图标。
- 对话数据。
- 对话播放器。
- 案件 intro/outro 文案。
- 成功/失败演出。
- 现场记录和结案摘要展示。

## 10. 推荐实施优先级

P0：

## 11. 给主题包装 AI 的简短任务说明

请基于《未登记现场》的保险异常调查设定，为现有格子推理游戏设计一套 UI 和原画包装。核心玩法不能改变：玩家通过公开现场定则推理，调查全部安全区域，标注全部异常区域，提交现场登记图。视觉和剧情不能暗示答案。需要输出：整体视觉风格、主角与搭档半身像方向、现场平面图 UI、现场定则卡片、登记面板、成功/失败弹窗、视觉小说对话框，以及首案《别按那个键》的关键美术资产方向。
